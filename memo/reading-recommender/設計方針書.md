読書推薦 app 技術選定・アルゴリズム設計書

1. 設計方針

新規 app は apps/reading-recommender として追加し、ローカル PC に常駐する SPA + API/worker 一体型アプリとして設計します。画面は Vite + React + TypeScript の SPA とし、同じ app 内の Fastify サーバが SQLite 読み込み、推薦更新、検索、embedding 生成、Android 向け push 通知を担当します。既存 monorepo は apps/* と packages/* を pnpm workspace として扱い、共有 SQLite は data/books.sqlite に置く前提なので、この構成は既存方針に合います。

重要な制約は、bookmeter app が data/books.sqlite の唯一の writer であることです。既存 README は、他プロジェクトが SQLite を read-only で開くべきこと、WAL モードにより 1 writer + 複数 reader の同時アクセスが安全であること、パスを BOOKS_DB_PATH で上書き可能にする方針を明示しています。したがって、推薦 app は data/books.sqlite を変更せず、推薦状態、検索インデックス、embedding、push 登録情報、設定値を data/reading-recommender.sqlite に保存します。

既定の実装は、主推薦 1 冊、副推薦 2 冊、関連書籍 5 件、検索結果 10 件とします。推薦では積読本を読みたい本より優先しますが、その優先度は score component として分離し、後から重みやロジックを差し替えられるようにします。embedding は外部 API とローカル実行を同じ interface で抽象化し、初期の推奨設定は精度を優先して OpenAI text-embedding-3-large、ローカル fallback は Transformers.js または Python sidecar の multilingual E5 系モデルとします。

2. 既存 SQLite と rowid に関する判断

現在の bookmeter schema は wish と stacked の 2 テーブルで、主キーは bookmeter_url、列は isbn_or_asin、book_title、author、publisher、published_date、所蔵情報、description です。登録日や読書メーター側の明示的な登録順カラムはありません。

rowid については、あなたの認識は「条件付きで近いが、そのまま履歴的な古さとして扱うのは危険」です。既存 bookRepository.ts は読み込み時に rowid asc で並べ、保存時には既存テーブルを一度全削除してから bookList.values() の順に再挿入しています。コメントにも「現在の remote 順を rowid に反映させる」とあります。つまり rowid は「SQLite に初めて入った順」ではなく、「最後にスクレイピングしたときの読書メーター上の remote order」です。

SQLite 自体は、ROWID を明示しない INSERT では通常、未使用の整数を自動で割り当てます。テーブルが空なら ROWID は 1 から始まり、通常は現在の最大 ROWID + 1 になります。ただし、これはあくまで挿入順を表す仕組みであり、現在の実装では毎回全削除・再挿入されるため、rowid は過去からの蓄積順ではなく最新スクレイプ時の表示順位になります。

スクレイパーは読書メーターの page=1 から順に巡回し、各ページで取得した DOM 要素の順に Map へ追加しています。wish も stacked もこの順序で bookList に入り、保存時に同じ順序で再挿入されます。したがって、読書メーターが「新しく追加した本を page 1 の上位に出す」仕様であるなら、rowid が大きいほど古い本とみなしてよいです。一方、読書メーター側の並び順が変更されたり、ユーザー操作で並び替えられたりする場合は、rowid は古さではなく表示順位になります。

設計上は source_rowid という名前ではなく、remote_rank として app 側に取り込みます。初期設定では remote_rank が大きいほど古いという仮定を使いますが、設定値 remoteOrderAgeDirection = "larger_is_older" | "larger_is_newer" | "disabled" を持たせます。さらに初回起動時に stacked の rowid 先頭 5 件と末尾 5 件を表示し、ユーザーが記憶と照合できる診断画面を用意します。この設計なら、あなたの仮説を活かしつつ、読書メーター側の表示仕様に依存しすぎない実装にできます。

3. 技術選定

3.1 フロントエンド

フロントエンドは Vite + React + TypeScript を採用します。Vite は SPA 向けの開発サーバと production build を持ち、React TypeScript テンプレートも公式に用意されています。既存 root の Volta 設定は Node 25.8.2 で、Vite の現在の Node 要件である 20.19+ または 22.12+ を満たします。 

React Router は導入してよいですが、画面数は少ないため過剰な framework は避けます。画面は「今週の推薦」「検索」「設定」「診断」の 4 ページで足ります。状態管理は server state が中心なので、TanStack Query を入れるか、最初は fetch + custom hook に留めます。推薦や検索の結果は API から取得し、ブラウザ側で embedding やランキングを直接計算しない設計にします。

3.2 API/worker

API と worker は Fastify + TypeScript を採用します。Fastify は Node.js の HTTP server として route を宣言しやすく、local PC 常駐の API、static file 配信、worker の orchestration を 1 つの process にまとめやすいです。Next.js は SSR や hosting 統合が必要な場合に有利ですが、この要件では local PC で SQLite を読む worker が中心なので、Vite SPA と Fastify API の分離が素直です。

production では Fastify が apps/reading-recommender/dist/client を static 配信し、同じ origin で /api/* を提供します。development では Vite dev server が SPA を配信し、API は Fastify の localhost:<apiPort> に proxy します。Android スマートフォンからアクセスするため、production 相当の local 常駐時は HTTPS origin が必要です。

3.3 SQLite アクセス

SQLite は better-sqlite3 を採用します。既存 bookmeter app でも better-sqlite3 と Drizzle ORM が使われており、pnpm workspace の allowBuilds にも better-sqlite3 が含まれています。better-sqlite3 は readonly option と fileMustExist option を提供しているため、source DB を new Database(booksDbPath, { readonly: true, fileMustExist: true }) で開く設計にできます。  

app DB は data/reading-recommender.sqlite とし、通常テーブルは Drizzle ORM で扱ってもよいです。ただし、FTS5 virtual table、trigger、migration、BLOB vector などは raw SQL のほうが明瞭です。したがって、schema.ts で型を定義し、migration は SQL ファイルで管理する折衷構成を推奨します。

3.4 Push 通知

Android スマートフォン向け push 通知は Firebase Cloud Messaging for Web を採用します。FCM の Web SDK は Push API 対応ブラウザで通知を受け取れますが、Service Worker を使うため HTTPS で配信される page が必要です。Android Chrome を対象にすれば、iPhone Safari の PWA 制約を考慮する必要はありません。

local PC 常駐で Android に push する場合、スマートフォンがアクセスする origin は HTTPS でなければなりません。実装上は Cloudflare Tunnel、Tailscale Funnel、ngrok、または LAN 内 HTTPS 証明書を使い、https://reading.example のような固定 origin を local Fastify に転送します。単なる http://192.168.x.x:<port> では Service Worker と FCM が成立しないため、これは実装前に必ず環境設定に入れる必要があります。

FCM 送信側は firebase-admin を Fastify process に組み込みます。Firebase Admin SDK は Node.js から特定 device の registration token に送信でき、複数 token への multicast 送信もできます。Web SDK 側は現在 Firebase Installation ID を推奨しつつ、registration token 取得 API も示しているため、DB には provider, token_type, target_id を保存し、将来の SDK 変更に備えます。 

3.5 Embedding

embedding は provider interface による差し替え式にします。初期設定は精度優先で OpenAI text-embedding-3-large、コストやオフライン要件が出た場合の fallback として local provider を使います。OpenAI の embedding docs では text-embedding-3-small が 1536 次元、text-embedding-3-large が 3072 次元で、dimensions parameter により短縮できるとされています。MTEB の掲載値でも text-embedding-3-large が text-embedding-3-small より高い性能として示されています。

local provider は 2 段階で考えます。軽量で TypeScript だけで済ませる fallback は @huggingface/transformers と Xenova/multilingual-e5-small です。Transformers.js はブラウザや JavaScript 環境で pretrained model を実行でき、ONNX Runtime を使います。Xenova/multilingual-e5-small は Transformers.js と ONNX に対応する feature-extraction model として公開されています。 

local でも精度を上げたい場合は、Python sidecar で sentence-transformers と intfloat/multilingual-e5-large-instruct を使う拡張を用意します。このモデルは 94 言語対応、1024 次元、0.6B parameters とされ、query には task instruction を付ける使い方が model card に示されています。ただし、長文は 512 tokens に truncate されるため、書籍説明文が長い場合は title/author/description を整理して入力する必要があります。

7. source DB 同期

source 同期は、起動時、手動更新、data/books.sqlite の変更検知、週次推薦前に実行します。source DB は wish と stacked をそれぞれ rowid asc で読み、bookmeter_url を key に union します。両方に存在する本は in_wish=1, in_stacked=1 とし、remote_rank_source は stacked を優先します。これは、積読本を読みたい本より優先する要件と一致します。

SELECT
  rowid AS source_rowid,
  bookmeter_url,
  isbn_or_asin,
  book_title,
  author,
  publisher,
  published_date,
  description
FROM stacked
ORDER BY rowid ASC;

同期時には content_hash = sha256(book_title, author, publisher, published_date, description, in_wish, in_stacked, remote_rank) を計算します。前回と content_hash が変わらない本は FTS と embedding を再生成しません。消えた本は book_snapshot から削除してもよいですが、推薦履歴の参照を保つため、実装上は last_seen_at を更新しないだけにして、検索対象から除外する方式も選べます。MVP では検索対象を last_scan_run_id = current_scan_run_id に限定します。

source hash は全対象本の bookmeter_url + content_hash + remote_rank を rowid 順に連結して計算します。ファイルの mtime だけでは、同じ内容の再生成やコピーを検知してしまうため、推薦更新の判断は source hash を正とします。

8. Embedding provider 設計

embedding は次の interface で固定します。

export type EmbeddingInputKind = "query" | "document";

export interface EmbeddingProvider {
  readonly providerId: string;
  readonly modelId: string;
  readonly dimension: number;

  embed(input: {
    texts: readonly string[];
    kind: EmbeddingInputKind;
  }): Promise<readonly Float32Array[]>;
}

OpenAI provider は text-embedding-3-large を既定にし、dimensions=1024 を推奨します。3072 次元のまま使うと精度面では最も素直ですが、書籍数が増えたときの保存量と計算量が増えます。OpenAI docs は dimensions parameter による短縮を想定しており、短縮しても概念表現を保つ設計であると説明しています。

Local Transformers provider は Xenova/multilingual-e5-small を既定候補にします。E5 系は query と document の prefix を分ける前提のモデルが多いため、provider 内で query には query: ...、document には passage: ... のような入力整形を閉じ込めます。これにより、上位の推薦エンジンは provider 差を意識しません。

Python sidecar provider は高精度ローカル用の拡張です。uv workspace を使って apps/reading-recommender/pyproject.toml を置き、Fastify から python -m reading_recommender_embeddings を child process として起動します。model は intfloat/multilingual-e5-large-instruct を候補にし、query に task instruction を付け、document には title/author/description を短く整形して渡します。model card は 512 tokens で truncate される制約を示しているため、長い説明文をそのまま投げず、先頭 800 から 1200 文字程度に制御します。 

embedding は provider と model が違えば同じ空間で比較できません。したがって、book_embedding の primary key に provider_id, model_id, dimension を入れ、app_setting.activeEmbeddingProvider が変わったら active provider の vector だけを検索・推薦に使います。

9. 検索アルゴリズム

検索は FTS5 + embedding + metadata の hybrid retrieval とします。FTS5 は文字列一致やタイトル一致に強く、embedding は「セキュリティ」と「暗号」「認証」「脆弱性」のような語彙差を吸収できます。片方だけに寄せると、タイトルや著者名検索、自然文 query、抽象的な分野検索のいずれかで品質が落ちるため、統合 ranking を標準にします。

FTS の rank は SQLite FTS5 の bm25() を使います。FTS5 の BM25 は値が小さいほど良い match で、列ごとの重み付けもできます。初期重みは book_title=6.0, author=3.0, publisher=1.5, description=1.0 とし、タイトルや著者の一致を説明文一致より強く扱います。

semantic rank は query embedding と document embedding の cosine similarity で計算します。OpenAI docs でも embedding-based search では query と document の cosine similarity を計算し、最も高いものを relevant とする方法が示されています。OpenAI embeddings は正規化済みで、cosine similarity は dot product で高速に計算できるとされています。

最終 ranking は Reciprocal Rank Fusion を使います。

rrf(book) =
  lexical_weight  / (k + lexical_rank)
+ semantic_weight / (k + semantic_rank)
+ metadata_weight / (k + metadata_rank)

既定値は k=60、自然文 query では semantic_weight=0.55, lexical_weight=0.35, metadata_weight=0.10 とします。ISBN、著者名、短い固有名詞のような query では lexical_weight=0.70, semantic_weight=0.20, metadata_weight=0.10 に切り替えます。切り替えは query classifier を置かず、最初は正規表現と文字種で判定します。

検索結果には rank だけでなく理由を返します。理由は「タイトルに一致」「説明文が近い」「同じ著者」「積読本」「remote_rank が古い」などに分解します。この理由生成は ranking と別関数にし、後から UI 表示を変えても ranking の再現性が崩れないようにします。

10. 推薦アルゴリズム

推薦は bundle policy と scoring engine を分けます。bundle policy は「今の推薦を継続するか、新しい bundle を作るか」を決めます。scoring engine は、新しい bundle が必要なときに候補本を順位付けします。この分離により、「推薦済み本が DB に残っている限り継続する」という要件を score より上位のルールとして固定できます。

10.1 Bundle policy

active bundle は主推薦 1 冊、副推薦 2 冊を持ちます。週次更新時に主推薦が source DB の wish ∪ stacked に残っていれば、主推薦は変えません。副推薦が source DB から消えていた場合だけ、主推薦を維持したまま副推薦を補充します。

主推薦が source DB から消えていた場合は、読了済みと判定します。この判定は「最新バージョンの SQLite ファイルにその本のエントリが含まれているか」という要件に従います。その後、残っている副推薦を候補に含めつつ、scoring engine で新しい主推薦 1 冊と副推薦 2 冊を選びます。副推薦を自動昇格させるか、全候補から再選定するかは設定可能にし、初期値は「副推薦を優先候補に含めて再選定」とします。

if activeBundle is null:
  createNewBundle(reason = "initial")

else if primaryBook is still in currentSourceSet:
  keepPrimary()
  refreshSecondariesIfMissing()
  sendWeeklyReminderIfScheduled()

else:
  markPrimaryCompleted(reason = "completed_by_db_absence")
  createNewBundle(reason = "primary_removed_from_source")
  sendNewRecommendationNotification()

前倒し推薦は、bookmeter app が SQLite を更新したあとに reading-recommender が source hash の変化を検知することで実行します。local PC 常駐なので、chokidar で data/books.sqlite を監視し、変更から数秒 debounce して source sync と bundle policy を走らせます。mtime だけで推薦を変えず、source hash が変わった場合だけ判定します。

10.2 Scoring engine

scoring engine は score component の加重和で作ります。

export interface ScoreComponent {
  readonly id: string;
  score(book: CandidateBook, ctx: RecommendationContext): ScoreContribution;
}

export type ScoreContribution = {
  value: number;              // 0.0 to 1.0
  explanation: string;
  debug: Record<string, unknown>;
};

初期 score は次の式にします。

score(book) =
  0.25 * listPriority(book)
+ 0.20 * remoteAge(book)
+ 0.20 * clusterCentrality(book)
+ 0.15 * semanticDensity(book)
+ 0.10 * metadataQuality(book)
+ 0.10 * novelty(book)
-       historyPenalty(book)

listPriority は積読本を読みたい本より優先します。初期値は stacked=1.0, wish=0.55, both=1.1 とします。これは設定で変更可能にし、将来「読みたい本を多めに崩す期間」「積読だけに絞る期間」のような方針変更を app_setting だけで行えるようにします。

remoteAge は rowid 由来の remote_rank を使います。remoteOrderAgeDirection="larger_is_older" の場合は remoteAge = (remote_rank - 1) / (max_rank - 1) とし、rank が大きいほど高くします。診断画面でこの仮定が誤りだと分かった場合は "larger_is_newer" に切り替え、式を反転します。表示順が信用できない場合は "disabled" にして重みを 0 にします。

clusterCentrality は、その本の周辺に関連本がどれだけ多いかを表します。各候補本について、embedding 類似度の上位 m=10 件の平均を計算し、関連分野の文献が複数登録されている本を少し優先します。これにより、「今週読むべき本はこれ。近い内容の本はこれ」という UI に自然につながります。

semanticDensity は clusterCentrality に近いですが、説明文やタイトルの意味的な密度を評価します。clusterCentrality は候補集合内で中心的な本を拾う役割、semanticDensity は局所的な類似本があることを拾う役割です。MVP では両者を統合してもよいですが、component を分けておくと後から重みを調整しやすくなります。

metadataQuality はタイトル、著者、出版社、説明文が揃っているほど高くします。説明文が空の本は embedding の品質が落ちるため、推薦理由を説明しにくくなります。ただし、説明文がないだけで良書を除外しないよう、重みは 0.10 に留めます。

novelty と historyPenalty は、同じ著者や同じ分野ばかりが連続することを防ぎます。直近 4 cycle に同じ著者が出ていれば penalty、同じ semantic cluster が続いていれば軽い penalty を加えます。ただし、active primary の継続ルールは penalty より優先されます。

11. 副推薦と関連書籍

副推薦 2 冊は、主推薦と同じ scoring engine で選びます。ただし、主推薦とほぼ同じ本ばかりになると「選択肢」にならないため、diversityPenalty を入れます。主推薦との cosine similarity が高すぎる本は副推薦から少し下げ、同じ分野だが別角度の本を残します。

secondaryScore(book) =
  baseRecommendationScore(book)
+ 0.10 * relatednessToPrimary(book)
- 0.20 * nearDuplicatePenalty(book, primary)

関連書籍は、副推薦とは別に、主推薦の下部に表示します。こちらは「選択肢」ではなく「近い内容の本」なので、主推薦との類似度を重視します。

relatedness(primary, book) =
  0.65 * cosine(embedding(primary), embedding(book))
+ 0.20 * lexicalSimilarity(primary, book)
+ 0.10 * sameAuthor(primary, book)
+ 0.05 * samePublisher(primary, book)

関連書籍は n=5 を初期値にします。UI では「同じ著者」「説明文が近い」「タイトル語が重なる」「同じ出版社」のような理由を出します。単に score を見せるより、なぜ近いと判断されたかを出したほうが、能動的読書の導線になります。

12. API 設計

API は SPA から呼ぶ public endpoint と、内部 worker から呼ぶ internal endpoint に分けます。local app でも tunnel で外に出す可能性があるため、更新系 endpoint は認証 token を要求します。

GET /api/recommendations/current
現在の主推薦、副推薦2冊、関連書籍、推薦理由を返す。

POST /api/recommendations/run
source sync と bundle policy を実行する。reason は manual, scheduled, source_changed のいずれか。

POST /api/recommendations/skip
現在の主推薦を skip し、新しい bundle を作る。読了扱いにはしない。

POST /api/recommendations/promote
副推薦を主推薦に昇格する。ユーザーが今週読む本を変更するための endpoint。

GET /api/search?q=...&limit=10
FTS5 と embedding を統合して検索結果を返す。

POST /api/index/rebuild
FTS と embedding を再構築する。provider 変更後に使う。

GET /api/books/diagnostics/row-order
rowid 先頭・末尾の本を返し、remote_rank と古さの仮定を確認する。

GET /api/settings
推薦曜日、時刻、重み、embedding provider、通知設定を返す。

PATCH /api/settings
設定を更新する。

POST /api/push/register
Android Chrome から push target を登録する。

POST /api/push/test
登録済み端末に test notification を送る。

GET /api/recommendations/current の response は UI がそのまま描画できる形にします。主推薦、副推薦、関連書籍、score breakdown、説明文、source list、Bookmeter URL を含めます。score breakdown は debug UI では詳細表示し、通常 UI では短い理由だけを表示します。

13. Scheduler と file watcher

local PC 常駐なので、scheduler は Fastify process 内に置きます。設定値として recommendationDayOfWeek, recommendationTime, timezone="Asia/Tokyo" を持ち、次回実行時刻を計算して setTimeout で予約します。process 再起動時には app_setting と recommendation_cycle を見て、未実行の予定が過ぎていれば即時実行します。

file watcher は data/books.sqlite と WAL 関連ファイルを監視します。bookmeter app の保存中に読み始めると不完全な瞬間を拾う可能性があるため、変更通知から 5 秒程度 debounce し、その後 source hash を計算します。source hash が変わった場合だけ、source sync、読了判定、前倒し推薦を実行します。

通知は 3 種類に分けます。週次時刻に主推薦が継続されている場合は reminder 通知を送ります。主推薦が DB から消えて新 bundle が作られた場合は new recommendation 通知を送ります。ユーザーが手動 skip または promote した場合は、原則として通知を送らず、画面上で完結させます。

14. Push 通知の実装詳細

Android Chrome で初回アクセスしたら、設定画面に「通知を有効化」ボタンを出します。ユーザー操作の中で Notification permission を要求し、FCM SDK を初期化し、登録識別子を /api/push/register に送ります。FCM Web は Push API 対応ブラウザと HTTPS を前提にするため、local PC 常駐でも Android からアクセスする URL は HTTPS にします。

Service Worker は public/firebase-messaging-sw.js に置きます。通知 payload には cycleId, bookmeterUrl, deepLink を入れ、本文に書名を出すかどうかは設定にします。読書傾向を lock screen や push provider に出したくない場合は、「今週の推薦本が更新されました」のような汎用文だけにします。

server 側は firebase-admin を使い、push_target の active token に multicast 送信します。Admin SDK は最大 500 registration tokens への multicast を 1 回で送れるため、Android スマートフォン、タブレット、desktop Chrome が複数登録されても同じ実装で扱えます。失敗 token は response の index と照合して disabled_at を更新します。

15. UI 設計

トップ画面は「今週読む本」を中心にします。主推薦カードにはタイトル、著者、説明文、積読本 または 読みたい本、推薦理由を表示します。カード下には「副推薦」として 2 冊を並べ、各本に「今週はこれを読む」ボタンを置きます。

主推薦の下部には「近い内容の本」を 5 件表示します。これは副推薦とは別で、主推薦を読む前後に参照したい本を見せる領域です。関連書籍は「説明文が近い」「同じ著者」「同じテーマのクラスタにある」などの理由と一緒に表示します。

検索画面では、query input と検索結果を表示します。結果にはタイトル、著者、リスト種別、説明文抜粋、検索理由を表示します。「セキュリティに関連する本はどれ？」のような自然文 query でも、embedding と FTS5 を併用して関連本を上位に出します。

設定画面では、推薦曜日、推薦時刻、timezone、積読優先度、remote_rank の解釈、embedding provider、通知許可、test notification を扱います。診断画面では rowid 先頭・末尾の本を表示し、「rowid が大きいほど古い」という仮定が正しいかを確認できるようにします。

16. セキュリティと運用

local PC 常駐でも、Android から HTTPS tunnel 経由でアクセスする場合は外部から到達可能な origin になります。したがって、APP_AUTH_TOKEN を設定し、更新系 API と設定 API には bearer token または cookie session を要求します。SPA の初回アクセス時に token を入力し、local storage ではなく session storage に保存する実装から始めます。

Firebase Admin の service account key は .env で path を指定し、repository には置きません。OpenAI API key も server 側だけで使い、browser bundle に出さないようにします。embedding provider が OpenAI の場合、書名、著者、説明文が外部 API に送られるため、設定画面で provider と送信対象を明示します。

通知本文に書名を含めるかは privacy setting にします。FCM は通知配送に外部 provider を使うため、秘匿性を重視するなら本文は汎用文にして、詳細はアプリ内で表示する設計が安全です。

17. テスト設計

unit test では、fixture SQLite を使って wish と stacked の union、rowid 取得、source hash、読了判定、bundle 継続、副推薦補充、score component の計算を検証します。特に重要なのは、「主推薦が source DB に残っている限り、週次更新でも主推薦が変わらないこと」と「主推薦が source DB から消えたら、読了済みとして新 bundle が作られること」です。

retrieval test では、少数の日本語書籍 fixture を作ります。「セキュリティ」「暗号」「認証」「数学」「統計」「臨床」「哲学」のような query に対して、期待する上位本を golden set として定義します。embedding provider は mock を使い、ranking の統合ロジックを再現可能にします。

integration test では、Fastify server を in-memory または temporary DB で起動し、GET /api/recommendations/current、POST /api/recommendations/run、GET /api/search を確認します。push は実送信せず、PushProvider interface を mock します。OpenAI と FCM は外部依存なので、CI では mock、手元では pnpm --filter @bookmaker/reading-recommender test:integration:external のような明示コマンドに分けます。

18. 実装順序

第一段階では、source DB read-only 読み込み、app DB migration、source sync、rowid 診断、主推薦 1 冊 + 副推薦 2 冊、FTS5 検索、トップ画面を実装します。この段階では embedding と push を mock または無効にし、推薦の継続判定を先に固めます。

第二段階では、OpenAI embedding provider、embedding cache、hybrid search、関連書籍表示を実装します。provider interface と book_embedding schema を先に固定し、OpenAI から local provider に切り替えても上位の推薦エンジンを変更しない形にします。

第三段階では、Android Chrome 向け FCM、Service Worker、HTTPS tunnel 前提の production 起動、週次 scheduler、file watcher を追加します。この時点で、週次通知と DB 更新後の前倒し推薦が動く状態になります。

第四段階では、local embedding provider を追加します。軽量 fallback は Transformers.js、精度重視の local mode は Python sidecar と multilingual E5 large instruct にします。最後に score component の debug 表示を充実させ、推薦理由と重み調整を UI から確認できるようにします。

19. 最終的な選定まとめ

この要件では、Vite + React + Fastify + better-sqlite3 + FTS5 + OpenAI/local embedding provider + FCM Web Push が最も整合的です。SPA は軽量に保ち、SQLite と scheduler と push 送信は local Fastify process に寄せます。source DB は read-only、app 状態は別 SQLite に分離し、既存 bookmeter app の責務を壊しません。

rowid は「古さそのもの」ではなく「最後のスクレイプ時の remote order」として扱います。現行実装は全削除・再挿入で remote order を rowid に反映しているため、読書メーターの表示が新しい順であることを確認できれば、rowid が大きい本を古い本として推薦に使えます。ただし、その仮定は設定可能にし、診断画面で検証できるようにします。

推薦ロジックは固定の巨大関数にせず、score component の合成にします。これにより、積読優先、remote_rank 優先、関連分野密度、履歴 penalty、metadata 品質を独立に調整できます。主推薦の継続判定は score より上位の bundle policy として実装し、要件どおり、推薦本が source DB に残っている限り翌週も継続します。
