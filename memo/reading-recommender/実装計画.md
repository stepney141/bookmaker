# 読書推薦 app 実装計画

## 実装の前提

reading-recommender は `apps/reading-recommender` に追加する TypeScript app として実装します。画面は Vite、React、TypeScript で作り、SQLite 読み込み、推薦更新、検索、通知送信は同じ app 内の Fastify server が担当します。`data/books.sqlite` は bookmeter だけが書き込む共有 SQLite なので、この app は `BOOKS_DB_PATH` を尊重して read-only で開きます。推薦履歴、検索 index、embedding cache、通知登録、設定は `data/reading-recommender.sqlite` に保存し、既存の writer 境界を保ったまま実装を進めます。

初期実装では、主推薦 1 冊、副推薦 2 冊、関連書籍 5 件、検索結果 10 件を既定値にします。推薦曜日、推薦時刻、timezone、score weight、`remote_rank` の解釈は `app_setting` に保存し、UI から変更できる構造にします。`remote_rank` は履歴的な登録順ではなく、最後のスクレイプ時の読書メーター表示順として扱います。このため、初回診断画面で rowid の先頭と末尾を確認し、`larger_is_older`、`larger_is_newer`、`disabled` のいずれかを選べるようにします。

主推薦の継続判定は score より上位の bundle policy として実装します。主推薦が最新の `wish ∪ stacked` に残っている限り、週次更新でも主推薦は維持します。主推薦が消えた場合は読了済みと判定し、残っている副推薦を優先候補に含めつつ、全候補から新しい bundle を再選定します。この方針により、読了判定の要件と推薦品質の調整可能性を両立させます。

## 第 0 段階: 基盤確認と仕様固定

第 0 段階では、実装に入る前に環境と未決仕様を固定します。設計方針書は技術選定と app DB schema を十分に具体化していますが、運用値と外部サービスの扱いは実装中にぶれやすいからです。ここで既定値を設定ファイルと DB migration に反映できる形にまとめます。次に、app scaffold と DB 同期を実装します。

- `pnpm-workspace.yaml` の `allowBuilds` が `better-sqlite3`、`esbuild`、Firebase 関連 package を許可していることを確認します。
- `apps/reading-recommender/package.json` の dependency version は実装時点の lockfile に固定し、`latest` のまま残さないようにします。
- `APP_AUTH_TOKEN`、`OPENAI_API_KEY`、`FIREBASE_SERVICE_ACCOUNT_PATH`、`FIREBASE_WEB_CONFIG_JSON`、`BOOKS_DB_PATH`、`READING_RECOMMENDER_DB_PATH` を環境変数として整理します。
- 既定 timezone は `Asia/Tokyo` とし、推薦曜日と推薦時刻は app setting の初期値で保持します。
- source DB から消えた本は app DB から物理削除せず、`last_seen_at` と `last_scan_run_id` で現在対象かどうかを判定します。

完了条件は、初期設定表、環境変数一覧、app DB path の決定が README または設定コードに反映されていることです。ここで外部サービスを有効化する必要はありません。

## 第 1 段階: app scaffold と source 同期

第 1 段階では、推薦に必要な最小の土台を作ります。Vite SPA、Fastify API、app DB migration、source DB read-only repository を先に実装すると、推薦、検索、UI を同じデータ契約の上で進められます。この段階では embedding と push 通知を無効または mock とし、SQLite 契約と推薦継続ルールを先に固めます。次に、推薦 engine と画面を載せます。

- `apps/reading-recommender` を pnpm workspace app として追加し、`package.json`、`tsconfig.json`、`tsconfig.server.json`、`vite.config.ts`、`index.html` を作成します。
- `src/server/index.ts` と `src/server/dev.ts` を作り、development では Vite proxy、production では Fastify static 配信を使います。
- `src/db/sourceBooks.ts` で `data/books.sqlite` を `new Database(path, { readonly: true, fileMustExist: true })` により開きます。
- `wish` と `stacked` を `rowid asc` で読み、`bookmeter_url` を key として union し、`in_wish`、`in_stacked`、`wish_rowid`、`stacked_rowid`、`remote_rank` を導出します。
- `src/db/appDb.ts` と raw SQL migration を実装し、`scan_run`、`book_snapshot`、`book_fts`、`recommendation_cycle`、`recommendation_item`、`recommendation_event`、`push_target`、`app_setting` を作成します。
- `source_hash` と `content_hash` を SHA-256 で計算し、内容が変わらない本は FTS と embedding cache の再生成対象から外します。
- `GET /api/books/diagnostics/row-order` と `POST /api/recommendations/run` を実装し、手動で source sync を実行できるようにします。

完了条件は、fixture SQLite と実 SQLite の両方から `wish ∪ stacked` を読み込めることです。`book_snapshot` は source DB に列を追加せずに更新され、`pnpm --filter @bookmaker/reading-recommender run type-check` と source sync の unit test が通る必要があります。

## 第 2 段階: 推薦 core とトップ画面

第 2 段階では、推薦要件の中心である「今週読む本」を実装します。推薦は bundle policy と scoring engine に分け、主推薦の継続判定を score から独立させます。この分離により、重み調整を後から変えても「DB に残る限り継続する」という要件が崩れません。次に、検索と診断画面を加えて能動的確認を実現します。

- `RecommendationEngine` を実装し、主推薦 1 冊と副推薦 2 冊を作成します。
- `bundlePolicy` を実装し、初回、週次継続、主推薦消失、副推薦補充、skip、promote を event として記録します。
- score component として `listPriority`、`remoteAge`、`clusterCentrality` の placeholder、`metadataQuality`、`novelty`、`historyPenalty` を分離します。
- 初期 score は設計方針書の重みに従い、embedding がない段階では `clusterCentrality` と `semanticDensity` を 0 または FTS 由来の近似値として扱います。
- `GET /api/recommendations/current`、`POST /api/recommendations/skip`、`POST /api/recommendations/promote` を実装します。
- React 側に「今週の推薦」画面を作り、主推薦、副推薦、推薦理由、Bookmeter URL、source list を表示します。
- 設定画面の最小版として、`remote_rank` の向き、推薦曜日、推薦時刻、timezone、件数設定を変更できるようにします。

完了条件は、主推薦が source DB に残っている限り週次実行でも変わらないことです。主推薦を fixture DB から消した場合は、読了 event が記録され、新しい bundle が作られる必要があります。UI では手動実行、skip、promote の結果が再読込なしで確認できる状態にします。

## 第 3 段階: FTS5 検索と診断 UI

第 3 段階では、embedding なしでも実用できる検索を作ります。FTS5 trigram は日本語の部分一致に強く、タイトル、著者、出版社、説明文を横断して検索できます。一方で 3 文字未満の query や ISBN、短い固有名詞は弱いため、LIKE fallback と query 種別ごとの重み切り替えを同時に入れます。次に、embedding による意味検索を重ねます。

- `book_fts` を `book_snapshot` の current rows から更新し、`book_title`、`author`、`publisher`、`description` を index 対象にします。
- `GET /api/search?q=...&limit=10` を実装し、FTS5 `bm25()` と metadata score を統合して返します。
- ISBN らしい query、2 文字以下の query、著者名らしい query は LIKE fallback を併用します。
- 検索結果には title、author、source list、description snippet、Bookmeter URL、検索理由を含めます。
- 検索画面を作り、自然文 query と短い keyword の両方で結果が確認できるようにします。
- 診断画面を作り、`wish` と `stacked` の rowid 先頭 5 件と末尾 5 件、現在の `remoteOrderAgeDirection`、source hash、最終 sync 時刻を表示します。

完了条件は、embedding provider が未設定でも検索と推薦画面が使えることです。日本語 query、ISBN、著者名、空 query、短すぎる query の test を作り、API が一貫した validation error または検索結果を返すことを確認します。

## 第 4 段階: embedding と hybrid retrieval

第 4 段階では、意味的な近さを使った推薦と検索を追加します。FTS5 は語彙一致に強い一方で、「セキュリティ」と「暗号」「認証」「脆弱性」のような語彙差を吸収しにくいからです。provider interface を先に固定し、OpenAI と local provider を同じ cache と ranking に接続します。次に、関連書籍表示を完成させます。

- `EmbeddingProvider` interface を実装し、`providerId`、`modelId`、`dimension`、`embed()` を固定します。
- `OpenAIEmbeddingProvider` を実装し、既定候補を `text-embedding-3-large` の `dimensions=1024` とします。
- `book_embedding` に provider、model、dimension、input hash、vector BLOB を保存し、provider が変わったときに cache が混ざらないようにします。
- query embedding と document embedding の cosine similarity を計算し、FTS rank、semantic rank、metadata rank を Reciprocal Rank Fusion で統合します。
- 主推薦に対する関連書籍を `relatedness(primary, book)` で 5 件返し、「同じ著者」「説明文が近い」「タイトル語が重なる」などの理由を生成します。
- embedding provider が未設定、API key がない、外部 API が失敗した場合は、FTS-only mode に degraded して UI を継続動作させます。

完了条件は、embedding cache の再利用、provider 変更時の再構築、FTS-only fallback が test されていることです。検索結果と関連書籍は、score と理由を分離して返し、UI の表示変更が ranking の再現性を壊さない状態にします。

## 第 5 段階: scheduler と file watcher

第 5 段階では、能動的確認に加えて、定期推薦と前倒し推薦を実装します。週次推薦は app process 内 scheduler で実行し、前倒し推薦は `data/books.sqlite` の変更検知と `source_hash` の変化で判定します。mtime だけでは同一内容の再生成やコピーで誤作動するため、source sync 後の hash を正とします。次に、push 通知を接続します。

- app setting から推薦曜日、推薦時刻、timezone を読み、次回実行時刻を計算します。
- process 起動時に未実行の予定が過ぎていれば、即時に `POST /api/recommendations/run` 相当の処理を行います。
- `chokidar` で `books.sqlite` と WAL 関連ファイルを監視し、変更後 5 秒程度 debounce して source sync を実行します。
- `source_hash` が変わった場合だけ、bundle policy を実行して読了判定と前倒し推薦を行います。
- scheduler と watcher は同じ `RecommendationService` を呼び、二重実行を避ける lock または serialized queue を持ちます。

完了条件は、週次時刻、手動実行、source 変更検知が同じ推薦状態遷移を使うことです。bookmeter 書き込み中の一時状態を拾っても、debounce と read-only retry により app が落ちないことを test します。

## 第 6 段階: Android Chrome 向け push 通知

第 6 段階では、受動的通知を追加します。FCM Web Push は Service Worker と HTTPS origin を必要とするため、local PC 常駐でも Android からアクセスする URL は HTTPS にします。Cloudflare Tunnel、Tailscale Funnel、ngrok、LAN 内証明書のどれを使っても app の責務は同じなので、server は origin の固定値と認証 token を設定から読むだけにします。次に、local embedding と debug UI を追加します。

- `public/firebase-messaging-sw.js` を追加し、通知 click で現在の推薦画面へ遷移できるようにします。
- 設定画面に通知有効化ボタンを置き、ユーザー操作の中で Notification permission と FCM token 取得を行います。
- `POST /api/push/register` と `POST /api/push/test` を実装し、`push_target` に provider、token type、target id、user agent、成功失敗時刻を保存します。
- server 側に `firebase-admin` を接続し、weekly reminder と new recommendation を送信します。
- 通知本文に書名を含めるかどうかを privacy setting にし、初期値は汎用文にします。
- 更新系 API、設定 API、通知 API は `APP_AUTH_TOKEN` による bearer token または session cookie で保護します。

完了条件は、Android Chrome で通知登録、test notification、週次 reminder、新推薦通知を確認できることです。FCM credentials がない環境では server が起動し、通知機能だけが disabled と表示される必要があります。

## 第 7 段階: local embedding と調整 UI

第 7 段階では、外部 API に依存しない検索と推薦を追加します。OpenAI provider は精度面で初期候補になりますが、書名、著者、説明文を外部 API に送るため、privacy と費用の観点から local provider を選べる必要があります。軽量 fallback を TypeScript で実装し、精度重視の local mode は Python sidecar として分けます。最後に、推薦理由と重み調整を可視化します。

- `LocalTransformersEmbeddingProvider` を追加し、`@huggingface/transformers` と multilingual E5 系 model を使います。
- 必要になった時点で `apps/reading-recommender/pyproject.toml` を追加し、Python sidecar provider を uv workspace に登録します。
- query と document の prefix、長文説明の切り詰め、model download cache を provider 内に閉じ込めます。
- score component の debug 表示を追加し、主推薦、副推薦、関連書籍の score breakdown を確認できるようにします。
- 設定画面から component weight と provider を変更できるようにし、変更後は index rebuild を促します。

完了条件は、OpenAI、Transformers.js、FTS-only の 3 mode が同じ API response schema を返すことです。provider 変更、model 変更、dimension 変更のどれでも cache key が分かれ、異なる embedding 空間の vector を混ぜないことを確認します。

## テスト計画

テストは推薦の状態遷移、source 同期、検索 ranking を優先します。この app の中心リスクは UI よりも、読了判定と推薦継続の誤動作にあるからです。外部サービスは mock し、OpenAI と FCM の実接続は明示的な external test に分けます。次に、段階ごとの完了条件に合わせて test suite を増やします。

- source sync test では、temporary SQLite に `wish` と `stacked` を作り、rowid、union、`remote_rank`、`source_hash`、`content_hash` を検証します。
- recommendation test では、初回推薦、主推薦継続、副推薦補充、主推薦消失時の新 bundle、skip、promote を検証します。
- retrieval test では、日本語 title、著者名、ISBN、短い query、自然文 query の FTS ranking と fallback を検証します。
- embedding test では、mock provider で cosine similarity と Reciprocal Rank Fusion の再現性を検証します。
- API integration test では、Fastify server を temporary app DB で起動し、current、run、search、settings、diagnostics を検証します。
- push test では、Firebase 実送信を行わず、PushProvider mock で token 無効化と通知 payload を検証します。

最小の CI 相当コマンドは `pnpm --filter @bookmaker/reading-recommender run type-check` と `pnpm --filter @bookmaker/reading-recommender run test` です。app が root script に登録された後は、`pnpm -r type-check` と `pnpm -r test` でも通る必要があります。外部接続を含む確認は、`test:integration:external` のような明示コマンドに分離します。

## 実装順序の要約

まず、workspace app、Fastify server、Vite client、app DB migration、source DB read-only 同期を作ります。次に、bundle policy と scoring engine を実装し、トップ画面で現在の推薦を確認できるようにします。その後、FTS5 検索、診断画面、embedding による hybrid retrieval、関連書籍表示を追加します。最後に、scheduler、file watcher、FCM Web Push、local embedding provider、debug UI を加えます。

この順序では、最初に読了判定と推薦継続という要件の中核を固定できます。外部 API、HTTPS tunnel、Push 通知、local model download は後段に回すため、導入失敗があっても推薦 app の基本機能は完成します。各段階は同じ app DB と API schema を拡張する形で進め、source DB の writer 境界を最後まで維持します。
